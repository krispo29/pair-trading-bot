━━━━━━━━━━━━━━━━━━━━

 1. เก็บข้อมูลไว้ใน Memory อย่างเดียว
— ปัญหาที่ร้ายแรงที่สุด

ตอนเริ่มทำ ผมเก็บทุกอย่างไว้ใน memory ของ server (positions ที่เปิดอยู่, config, scan results ฯลฯ) พอ server restart ทุกอย่างหายหมด

ผลที่ตามมาคือ ระบบตรวจจับ orphan (ออเดอร์กำพร้า) มันเห็นว่า “มี position บน OKX แต่ไม่มีใน memory ของเรา” ก็เลยสั่งปิดออเดอร์ที่ถูกต้องทิ้ง สูญเสียเงินจริง

วิธีแก้: ย้ายทุกอย่างไป Database ตั้งแต่แรก ทุก state ที่เกี่ยวกับเงินห้ามอยู่ใน memory เด็ดขาด ต้องมี reconciliation (เช็ค DB กับ Exchange ให้ตรงกัน) ทุก 5 นาที และ Orphan Detection ต้องเป็น “แจ้งเตือนเท่านั้น” ห้าม auto-close

━━━━━━━━━━━━━━━━━━━━

 2. ไม่ตรวจสอบข้อมูลก่อนส่งคำสั่งไป Exchange
— ออเดอร์ขนาด Infinity

ตอนส่งคำสั่งเปิดออเดอร์ไป OKX ไม่ได้ตรวจว่าเหรียญมี Futures contract จริงไหม ไม่ได้ตรวจว่าราคา > 0 ไหม ผลคือ สูตรคำนวณขนาดออเดอร์ได้ค่า = Infinity (เพราะหารด้วย 0) ส่งไป OKX แล้ว Leg A (ขาแรก) สำเร็จ แต่ Leg B (ขาที่สอง) ล้มเหลว กลายเป็น orphan ขาเดียวค้างบน exchange

แถมตอนนั้นยังไม่มี rollback ถ้า Leg B fail ก็ไม่ได้ปิด Leg A คืน ปล่อยค้างไป

วิธีแก้: Validate ทุกอย่างก่อนส่ง (instrument มีจริง, price > 0, sz > 0, sz ไม่ใช่ Infinity) ถ้า Leg B fail ต้อง rollback ปิด Leg A ทันที ส่งคำสั่งปิดแล้วต้องตรวจซ้ำว่าปิดจริง (post-close verification)

━━━━━━━━━━━━━━━━━━━━

 3. ไม่มีระบบป้องกันการทำซ้ำ
— เปิดออเดอร์ซ้ำ, สัญญาณซ้ำ, แจ้งเตือนซ้ำ

ระบบไม่มี deduplication ตั้งแต่เริ่ม ผลคือ Scanner ส่งสัญญาณคู่เดิมซ้ำทุกรอบ scan, Auto-Trade เปิดออเดอร์คู่เดิมซ้ำ, Telegram ส่ง alert ซ้ำ

สุดท้ายต้องเพิ่ม dedup ถึง 5 ชั้น:
ชั้น 1 — เช็ค memory ว่าคู่นี้เปิดอยู่แล้วไหม
ชั้น 2 — เช็ค DB ว่าคู่นี้มี record open อยู่ไหม
ชั้น 3 — เช็ค OKX จริงว่ามี position อยู่ไหม
ชั้น 4 — เช็ค cooldown ว่าเพิ่งปิดไปไม่ถึง 1 ชม.ไหม
ชั้น 5 — เช็ค groupId ว่าออเดอร์ชุดนี้ปิดแล้วหรือยัง

วิธีแก้: ออกแบบ dedup ตั้งแต่ design phase ทุก signal, ทุก order, ทุก notification ต้องมี unique ID + cooldown + idempotency check

━━━━━━━━━━━━━━━━━━━━

 4. ไม่มี Safe Entry Zone
— เปิดออเดอร์แล้วโดน Stop Loss ทันทีใน 6 วินาที

คู่ SATS/RSR ถูกเปิดที่ Z-Score = 3.607 ซึ่งเกิน Stop Loss threshold (3.5) อยู่แล้ว ผลคือเปิดแล้วโดน SL ปิดทันทีใน 6 วินาที เกิดซ้ำ 2 ครั้งติดต่อกัน

พอตรวจย้อนหลังพบว่ามีออเดอร์ที่เปิด-ปิดเร็วผิดปกติถึง 48 รายการ ต้องลบออกจาก DB ทั้งหมด

วิธีแก้: สร้าง Safe Entry Zone ห้ามเปิดถ้า Z-Score ใกล้ SL เกินไป (เช่น ถ้า SL = 3.0 ห้ามเปิดเมื่อ Z > 2.5) เพิ่ม Grace Period 5 นาที ไม่ trigger SL ทันทีหลังเปิด

━━━━━━━━━━━━━━━━━━━━

 5. Hot Reload สร้าง Instance ซ้ำ
— ทุกอย่างทำงานซ้อนกัน

ตอน dev ใช้ tsx watch (auto-reload เมื่อแก้โค้ด) ปัญหาคือมันสร้าง instance ใหม่ทุกครั้งที่ไฟล์เปลี่ยน แต่ไม่ cleanup ตัวเก่า ผลคือ Scheduler ทำงาน 2-3 ตัวพร้อมกัน, Telegram Bot รับ message เดียวกัน 2-3 ครั้ง, Orphan Detection แจ้งเตือนซ้ำ

วิธีแก้: ใช้ globalThis เก็บ reference ของทุก interval/timeout/polling แล้ว cleanup ก่อนสร้างใหม่ทุกครั้ง ระบบที่มี background jobs ต้องจัดการ lifecycle ตั้งแต่แรก

━━━━━━━━━━━━━━━━━━━━

 6. ตั้งค่าจากการคาดเดา ไม่ได้ Backtest
— Config อยู่อันดับ 421 จาก 540

ค่า config เริ่มต้น (Z-Entry = 1.5, DI Filter = 17.7 ฯลฯ) ถูกตั้งจากการคาดเดา ไม่ได้มาจาก backtest จริง

พอรัน full backtest ทดสอบ 540 ชุด filter combinations ย้อนหลัง 3 ปี พบว่า config ที่ใช้อยู่ได้อันดับ 421 จาก 540 (ล่างสุด 22%) ปัญหาหลักคือ DI Filter = 17.7 เข้มงวดเกินไป กรองคู่ที่ดีออกไปเกือบหมด

วิธีแก้: Backtest ก่อนตั้งค่า default เสมอ ไม่ใช่คาดเดาแล้วค่อย backtest ทีหลัง และต้อง re-optimize เป็นระยะ (ทุก 1-2 สัปดาห์)

ถ้าจะสร้างระบบเทรดอัตโนมัติ
5 สิ่งที่ต้องทำตั้งแต่วันแรก

 1. DB-first — ห้ามเก็บ state ที่เกี่ยวกับเงินไว้ใน memory
 2. Validate ทุกอย่าง — ก่อนส่งคำสั่งไป Exchange ตรวจให้ครบ
 3. Dedup ตั้งแต่ design — ทุก signal, order, notification ต้องมี unique ID
 4. Backtest ก่อนตั้งค่า — อย่าคาดเดา ให้ data บอก
 5. Defensive by default — ถ้าไม่แน่ใจ ให้ไม่ทำ ดีกว่าทำแล้วผิด

ระบบเทรดต่างจากเว็บทั่วไปตรงที่ ทุก bug มีต้นทุนเป็นเงินจริง เปิดออเดอร์ซ้ำ 1 ครั้ง เสียเงิน $500 ปิดออเดอร์ดีผิดพลาด 1 ครั้ง เสียโอกาสกำไร $1,000

ถ้าออกแบบให้ defensive ตั้งแต่แรก คาดว่าจะลด bug ได้ 70-80% และประหยัดเวลาพัฒนาได้ 40%

หวังว่าจะเป็นประโยชน์กับคนที่กำลังจะสร้างระบบเทรดครับ
